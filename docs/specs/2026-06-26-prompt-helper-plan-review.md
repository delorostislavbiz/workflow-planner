# Codex Adversarial Review — план имплементации Prompt-помощника

Target: working tree diff (`docs/specs/2026-06-26-prompt-helper-plan.md`)
Verdict: **needs-attention**

No-ship: план оставляет непокрытые guardrails спека и содержит verify, который можно формально пройти без доказательства корректного поведения.

## Findings

- **[high] Не запланированы краевые guardrails из §17** (`...-plan.md:101-103`)
  Плановая самопроверка заявляет покрытие §5–§20, но фактическая карта пропускает §17. Из дизайна там приходят дорогие failure modes: отсутствие доступа к вебу/репо/секретам, задачи с актуальными источниками, high-stakes, слишком широкий проект, запрос вложенных субагентов. Без отдельного шага эти правила легко не попадут ни в prompt-helper, ни в тесты.
  *Recommendation:* отдельный шаг для §17 в Phase 1 (секция Edge cases в `reference/prompt-helper.md` с проверкой всех bullets §17); 2–3 фикстуры на no-access / current-sources / high-stakes / nested-subagents; обновить coverage map.

- **[high] Grep-verify для нормативного правила нереалистичен** (`...-plan.md:12-13`)
  Verify предлагает grep по широким токенам «3+»/«источник» и ожидает найти правило один раз как норматив. В реальном `applicability.md` уже есть не-нормативные упоминания 3+ (fit criteria, examples, probe questions). Реализация либо не пройдёт проверку, либо начнёт удалять полезные примеры ради счётчика grep.
  *Recommendation:* заменить broad grep на проверку уникального якоря/заголовка нормативного правила (напр. `Shared-source threshold rule`); остальные упоминания — только как ссылки на этот anchor. В verify различать normative definition и explanatory references.

- **[medium] Чек-лист можно реализовать как список заголовков, нарушив UX-спек** (`...-plan.md:31-32`)
  Шаг требует 10 пунктов и точечно проверяет п.6/п.8/п.10, но §12 требует для каждого пункта простой вопрос + строку «зачем». Текущий verify пропустит тяжёлый внутренний чек-лист без novice-friendly вопросов.
  *Recommendation:* в 1.5 — формат каждого из 10 пунктов включает `question` и `why`; verify проверяет все 10 на наличие обоих элементов и содержательность, а не только количество и три спец-подпункта.

- **[medium] Валидация не воспроизводима и не оставляет ревьюируемого доказательства** (`...-plan.md:72-73`)
  Фаза 6 говорит «симуляция/прогон» и кладёт pass/fail только во внешний полигон. Для markdown-скила слишком слабый test contract: нет точного протокола запуска, формата transcript/result, версии файлов, consent/no-consent сценариев.
  *Recommendation:* в `tests/prompt-helper/` — runbook/rubric protocol: точный формат прогона, transcript template, pass/fail evidence fields, обязательные negative cases на отсутствие согласия, ссылка на сохранённые результаты полигона в финальной сверке.

## Next steps
Перед имплементацией усилить план по §17, проверкам нормативного правила, формату чек-листа и воспроизводимости тестов.
