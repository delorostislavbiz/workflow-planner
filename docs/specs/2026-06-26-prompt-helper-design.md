# Дизайн: «Prompt-помощник» — дополнение к workflow-planner

- **Дата:** 2026-06-26
- **Версия:** v2 (учтено критическое ревью Codex — см. `2026-06-26-prompt-helper-design-review.md`)
- **Статус:** дизайн пересмотрен, на повторную вычитку
- **Язык документа:** русский (внутренний дизайн-артефакт; рантайм-файлы скила остаются на английском)

---

## 0. Мини-глоссарий (термины держим строго)

- **workflow-planner** — этот скил Claude Code.
- **планировщик** — существующий поток скила: гейт → план → JS-скрипт.
- **Workflow tool / Dynamic Workflow** — рантайм, который запускает JS-скрипт с агентами (`parallel`/`pipeline`).
- **target-workflow (целевой воркфлоу)** — тот воркфлоу, ПОД который мы пишем промпт. Помощник его НИКОГДА не запускает.
- **review-workflow (проверочный воркфлоу)** — отдельный мультиагентный прогон промпта на дыры (этап 2).
- **leaf-agent (агент-лист)** — агент внутри воркфлоу без вложенных субагентов. Дальше — «leaf-agent».
- **Prompt-помощник** — это дополнение.

## 1. Проблема и мотив

`workflow-planner` **принимает** готовую задачу, но почти не **помогает её сформулировать**. «Нормальный на вид» промпт прячет дыры (переобещанная автономность, непроверяемые критерии, фальшивая параллель, leaf-agent без данных). Отдельно: для многих пользователей мультиагентный воркфлоу — новое и непонятное явление. Цель двойная: **корректность** промптов и **популяризация** (новичок понимает, что это и когда нужно).

## 2. Цель и не-цели

**Цель.** Помощник внутри `workflow-planner`, ведущий пользователя (упор на новичка) от идеи — или от черновика — к корректному промпту для target-workflow, начиная с честного «нужен ли тебе воркфлоу вообще».

**Не-цели.**
- Не запускает target-workflow.
- Не запускает никакую проверку молча — только по согласию (см. §9, три режима запуска).
- Не создаёт `.claude/agents` (это agent-constructor).
- Не заменяет поток гейт→план→скрипт; встаёт перед ним для размытых входов.

## 3. Пользователь и два режима подачи

Оба, **с упором на новичка**. Логика одна, подача — двумя режимами (Codex #14):
- **novice** — объяснение одной фразой, один вопрос за раз, примеры простыми словами.
- **concise** — помощник сам предлагает наиболее вероятный паттерн и список недостающих слотов одним компактным блоком; не гоняет по одному вопросу.

Режим выбирается по сигналам (новичок/опытный, явная просьба «покороче») и переключается по запросу.

## 4. Выбранный подход

**Гибрид (подход 3):** гейт первым → развилка «рецепт / интервью» → сборка → опциональная проверка. Берёт гибкость интервью и скорость рецептов; лидирующий гейт лечит склонность рецептов тянуть в воркфлоу там, где надо линейно.

**Упаковка решена:** живёт ВНУТРИ `workflow-planner` как отдельный поток. Пользовательский API — это **маршрутизация входов** (§5), а не «отдельный скил против режима»; маршрутизацию фиксируем сейчас (Codex #1).

## 5. Маршрутизация входов (контракт, Codex #1)

Развилка вверху `SKILL.md`. **Явная команда пользователя всегда сильнее автоопределения.**

| Вход | Маршрут |
|------|---------|
| Готовая чёткая задача (сформулировано, ясны единицы/цель) | текущий поток: гейт → план → скрипт |
| «Помоги сформулировать» / идея / «не знаю, нужен ли воркфлоу» | **build-path** Prompt-помощника |
| «Проверь мой черновик» / вставлен почти готовый промпт | **draft-audit** Prompt-помощника |
| Явная команда (например `prompt`, `audit`) | соответствующий маршрут принудительно |

Граничные случаи автоопределения трактуем консервативно: не перехватывать готовый промпт в полную пересборку (для него — draft-audit, не build-path).

## 6. Архитектура — два потока

**build-path (из идеи):**
```
0 Режим/язык  →  1 Гейт  →(линейно: рекомендация+короткий план+СТОП)
                          →(подходит)→ 2 Рецепт | Интервью
   →  3 Чек-лист  →  4 Сборка  →  5 Проверка (опц.)  →  6 Выдача
```
**draft-audit (из черновика, Codex #5):**
```
0 Режим/язык → 1 Гейт-экспресс (нужен ли воркфлоу?)
   → A Проверка черновика по чек-листу и граблям
   → B Patch-рекомендации: оставить / исправить / добавить / убрать
   → (полная пересборка — ТОЛЬКО по просьбе)
```

Принципы: гейт первым; владение фактами по §7; модель состояния §8; ничего не стартует без согласия §9.

## 7. Компоненты и владение (Codex #9)

Чёткие владельцы факта, чтобы не пересекались:
- **gate** — решает ТОЛЬКО `workflow / no-workflow / unclear`. Ничего не собирает.
- **pattern** — предлагает форму и **минимальные** pattern-слоты.
- **checklist** — валидирует полноту; **не задаёт уже закрытые вопросы**.
- **assembler** — только формулирует промпт, новых решений не принимает.

Реализация — markdown-инструкции скила (как у самого `workflow-planner`).

## 8. Модель состояния (Codex #4)

`reference/prompt-helper.md` обязан вести внутреннюю таблицу состояния (это «память» markdown-скила между репликами):

`mode` · `output_language` · `gate_decision` · `pattern_id` · `known_facts` · `assumptions` · `missing_blockers` · `open_questions` · `risk_flags` · `next_action`.

Правила:
- **Не спрашивать повторно**, если слот уже заполнен.
- **Не собирать финальный промпт**, пока есть незакрытый blocker-слот без явного допущения.
- Каждое допущение фиксируется в `assumptions` и проговаривается пользователю.

## 9. Три режима запуска (Codex #2 — снимает противоречие)

- **inline-check** — один ассистент, без агентов. Только после вопроса-согласия. **Это вся проверка в MVP.**
- **review-workflow** — отдельный проверочный воркфлоу с агентами. Только после отдельной явной фразы согласия. **Этап 2, не MVP.**
- **target-workflow** — воркфлоу, под который пишется промпт. Помощник **никогда** его не запускает.

«Ничего не запускаем без явного согласия» теперь однозначно: молча — ничего; inline-check — после вопроса; review-workflow — после отдельного согласия.

## 10. Гейт + decision-matrix (Codex #7, #23)

Канон гейта живёт в `reference/applicability.md` (его и расширяем матрицей), помощник лишь подаёт его человеческим языком и ссылается — правило не дублируем по файлам.

Базовое правило (есть): мысленно убрать параллельность/поток — если ничего не теряется, задача линейная.

Матрица сигналов (добавляем в applicability.md):
- **fit:** независимые единицы; нужны независимые экспертизы / adversarial-проверка; проверяемые артефакты; общий синтез; высокая цена ошибки.
- **no-fit:** 1–2 шага; общий mutable-файл без изоляции; цель неясна; нет проверяемого результата; дешевле линейно.
- **caution:** нужны доступ/секреты; destructive-изменения; динамический список задач; сильная зависимость от человека.

Правило «3+ кусков из общего источника → фаза-якорь; 2 → линейно» — нормативно в applicability.md, ссылаемся (Codex #23).

## 11. Развилка + рецепты (Codex #8, #14, «не вываливать меню»)

- **novice:** помощник предлагает ОДИН наиболее вероятный паттерн («похоже на Веер — подходит?») + «не подходит». Полное меню — не вываливаем.
- **concise:** показываем компактный список форм сразу.

**Рецепт = карточка:** *когда подходит* · *форма (фазы + примитив)* · *слоты* · **реализуемость** (Codex #8 — честно про ограничения).

Стартовый набор (5 + «сложнее»):
1. **Веер** → `parallel` → сборка.
2. **Конвейер** → `pipeline`.
3. **Общий источник** → фаза-якорь → `parallel`.
4. **Исследование с проверкой** → веер поисков → дедуп → проверка → синтез.
5. **Панель вариантов** → `parallel` попытки → оценка → синтез.
6. **«сложнее»** → свободное интервью.

**Реализуемость (что писать в каждой карточке):** какие фазы статичны; где список единиц должен быть известен ДО `parallel`; циклы/retry/дедуп/синтез **допустимы на уровне оркестратора** (JS-скрипт), а вот **вложенных субагентов нет** (агенты — leaf); чего из-за этого нельзя обещать. (Точность важнее перестраховки: «нет вложенности» ≠ «нет циклов».)

Рецепт — каркас, не клетка: не лезет → откат к интервью. Оба пути сходятся в чек-лист §12.

## 12. Чек-лист «правильного промпта» (сердце)

Внутренний контроль качества, не анкета. Вытягивается по ходу; рецепт часть закрывает; наружу — только нужные вопросы. Каждый пункт = простой вопрос + строка «зачем».

1. **Цель и результат** — чего хотим, как выглядит «готово».
2. **Что отдать на выходе** — отчёт/файлы/текст (по умолчанию — описание задачи для планировщика).
3. **Единицы параллелизма** — что разом, сколько, однотипны ли.
4. **Зависимости и порядок** — что чего ждёт, где барьеры.
5. **Общий источник правды?** — ссылка на нормативное правило (см. §10).
6. **Данные в промпт leaf-agent (Codex #11)** — промпт должен содержать: цель, входные артефакты, **пути/ссылки**, ограничения, критерии результата, формат ответа. Полные данные — через явно доступные файлы или anchor-артефакт; **нельзя** «посмотри контекст выше». (Не «вали всё в текст» — иначе гигантские промпты.)
7. **Verify по содержанию** — у каждой единицы своя проверка по сути, не по количеству.
8. **Автономность по уровню риска (Codex #10, решение пользователя)** — read-only → автономно до конца; write/destructive/платное/внешнее → подтверждение перед опасным действием; риск неясен → один уточняющий вопрос или зафиксировать ограничение «опасные действия не выполнять». Плюс: что считается аварией.
9. **Критерии приёмки** — как понять, что готово и хорошо, и кто проверяет.
10. **Границы записи и среда (Codex #12)** — отдельный мини-чек-лист: какие файлы read-only; что меняет каждый агент; кто делает merge/synthesis; что при конфликте; нужны ли `worktree`/ветки (+git); какие команды запрещены. Запись в одни файлы → `isolation:'worktree'` + git. Вложенных субагентов нет.

## 13. Сборщик

Превращает заполненный чек-лист в цельный промпт на `output_language`, по лёгкому скелету. Решений не принимает (см. §7).

## 14. Проверка на дыры + repair-loop (Codex #2, #13)

**MVP — только inline-check** (по предложению, не молча). Глубокий review-workflow — этап 2 (§9, §21).

**Грабли:** переобещанная автономность; непроверяемые критерии; agent-washing; фальшивая параллель / пропущенный барьер; общий источник без якоря; нехватка данных leaf-agent (п.6); verify по количеству; пропущенный шаг; опасные действия без подтверждения; пересекающиеся записи в файлы.

**Repair-loop (Codex #13):**
- дыра чинится допущением → предложить текст допущения (в `assumptions`);
- нужен факт → задать ОДИН вопрос;
- дыра блокирует воркфлоу → вернуться к гейту или выдать no-go;
- после правки пересобрать только затронутые части промпта.

Находки ранжированы (блокер/важно/мелочь). Реальный target-workflow не запускается.

## 15. Линейный исход и настойчивость (Codex #3)

Точные переходы:
- по умолчанию линейный исход = линейная рекомендация + короткий план + **СТОП**;
- если пользователь **явно** просит воркфлоу после предупреждения → продолжаем, но помечаем промпт «workflow выбран вопреки рекомендации»;
- «даём оба» — только по явной просьбе, не автоматически.

## 16. Размытый ввод — условие выхода (Codex #15)

После двух уточняющих вопросов:
- сформулировать допущения;
- спросить «собрать черновик по этим допущениям или уточнить ещё?»;
- если нет цели / результата / единиц параллелизма — **не выдавать** финальный промпт как готовый.

## 17. Краевые случаи (Codex #16, расширено)

- Настаивает на воркфлоу при линейной задаче → §15.
- «Сделай максимально автономно» для destructive/write → §12 п.8: подтверждение перед опасным шагом.
- Нет доступа к вебу/репо/секретам/инструментам → пометить ограничение, не подставлять молча.
- Задача требует актуальных источников → отметить как риск/ограничение.
- Слишком широкий проект без границ → декомпозировать, брать первый кусок.
- Агенты должны менять пересекающиеся файлы → §12 п.10 (worktree/ownership).
- High-stakes (юр./мед./фин.) → повышенная осторожность, явные подтверждения.
- Просит вложенных субагентов → честно: их нет; предложить плоскую форму или маршрут в agent-constructor.
- Размытый/голосовой ввод → интерпретировать по смыслу, проговорить допущения.

## 19. Тестирование (Codex #6, #17, #19)

Скил — на промптах. Фикстуры и rubric — **в репо**, полигон `WORKFLOW-PLANNER-TEST` — только для прогонов.

Для каждого теста заданы: **вход · ожидаемый маршрут · 2–5 ключевых утверждений · запретные действия · критерий pass/fail.**

Примеры (≥4, Codex #19) в `examples.md`:
1. идея → no-workflow линейный план (гейт сказал «не нужен», СТОП);
2. идея → рецепт Веер (распознал, собрал, прошёл чек-лист);
3. черновик → draft-audit + patch (нашёл дыры, дал оставить/исправить/добавить/убрать);
4. нестандартная → свободное интервью.

Дополнительно: общий источник (3+) → фаза-якорь; размытый голосовой ввод → условие выхода §16; **языковой тест (Codex #17): английский/смешанный ввод → промпт на языке ввода.**

**Критерий прохождения:** (a) гейт верен; (b) маршрут верен; (c) промпт удовлетворяет чек-листу §12; (d) проверка ловит заложенные дыры; (e) запретные действия не выполнены (ничего не запущено без согласия); (f) **язык совпадает с языком ввода** (не навязан русский).

## 20. Тон — операциональные правила (Codex #22)

Вместо «тепло/не проповедь»: не стыдить; не спорить дольше одного сообщения; причину объяснять одним предложением; всегда давать следующий практический шаг.

## 21. Этап 2 / отложено

- **review-workflow** — глубокий мультиагентный прогон промпта (как прогон 2026-06-26); отдельный этап, явное согласие.
- **Codex-sidecar** — перенос дополнения в `plugins/workflow-planner-codex` (своя упаковка/синтаксис).
- **Расширение набора рецептов** — кандидат «обзор по измерениям → проверка находок».
